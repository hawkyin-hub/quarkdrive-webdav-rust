## 8. P1-2 二次修复 — flush() 不应强制上传 .DS_Store/metadata 文件（2026-07-08）

### 现象
用户报告：**拖放上传文件失败（Error -36/-43），云盘内部移动文件也失败**。
截图显示 Finder 弹出三个错误对话框：clash_meta 返回 -36（I/O error）、其他文件返回 -43（File not found）。

### 根因
P1-2 的初次实现过于激进。`flush()` 中对**所有 `write_mode=true && !is_uploading`** 的文件都调用 `prepare_for_upload()`，包括：
1. **`.DS_Store`** — Finder 每次打开 WebDAV 目录时都会尝试创建/更新，fid 为空
2. **`._*` AppleDouble 资源 fork 文件**
3. **Finder 的 vnode probe（0 字节 PUT 探测）**

这些文件的 `self.file.fid` 为空 → `prepare_for_upload()` 设置 is_uploading 后进入 `do_flush()` → 某些路径返回 `NotFound` → `flush` 失败返回 Err → WebDAV 服务端报错 → **webdavfs_agent 向 Finder 返回 Error -36/-43 → 整个拖放操作中止**。

关键日志证据：
```
ERROR quarkdrive_webdav::vfs: file: flush failed file_id= file_name=.DS_Store error=NotFound
WARN  quarkdrive_webdav::vfs: fs.open: parent directory not found even after dfs parent=/媳克上传文件
ERROR quarkdrive_webdav::webdav: HTTP serve error: connection error
```

### 改动
**文件**：`quarkdrive-webdav/src/vfs.rs` 方法 `fn flush()`

**Before（初次 P1-2，有 bug）**：
```rust
if !self.upload_state.is_uploading {
    if !self.write_mode {
        self.upload_state.flush_count = self.upload_state.flush_count + 1;
        return Ok(());
    }
    // 对所有 write-mode 文件都走 upload 路径 ← 太激进
    self.prepare_for_upload().await?;
}
```

**After（二次修复）**：
```rust
if !self.upload_state.is_uploading {
    if !self.write_mode {
        self.upload_state.flush_count = self.upload_state.flush_count + 1;
        return Ok(());
    }
    // 三重守卫：只对真正的用户空文件走 upload 路径
    let name = self.file.file_name.as_str();
    if is_macos_metadata_file(name) || self.file.fid.is_empty() || self.parent_file_id.is_empty() {
        debug!(file_name = name, "flush: skipping metadata/empty-fid write (no upload)");
        self.upload_state.flush_count = self.upload_state.flush_count + 1;
        return Ok(());
    }
    // Genuine user-created empty file (e.g. touch blank.txt):
    self.prepare_for_upload().await?;
}
```

## 9. 子目录移动空 fid → HTTP 400（rename move 分支，2026-07-08）

### 现象
用户报告：第一个拖放上传成功，**第二个没报错但实际没成功**；**在挂载盘内拖动文件到子目录失败**。
修复前日志证据：
```
ERROR ... move file failed from=/verify_clean_...bin to=/视频附件文件/...bin error=HTTP status client error (400 ) for url (.../file/move...)
```

### 根因
`rename()` 的 move 分支（源/目标不同父目录）调用 `get_file(from)` 取源文件，再
`move_file(&file.fid, &to_parent_file.fid)`。
- `rename → get_file → find_in_cache` 读的是父目录 `dir_cache`（只缓存云端真实列表，真实 fid）。
- 但上传 commit 后的窄"最终一致"窗口内，云端列表可能仍以**空 fid** 形式暴露该文件；此时
  `file.fid` 为空 → `move_file("")` 命中 Quark `/1/clouddrive/file/move` → **HTTP 400**（或静默
  no-op，文件留在原地，Finder 报"没成功"）。这正是用户"拖动到子目录失败"且无报错的根因。
- 注：`read_dir` 会把 `active_writes`（fid=`""`）合并进目录列表，但该合并结果**不写回** `dir_cache`；
  真正触发 400 的是云端列表短暂返回空 fid 的窗口，而非 active_writes 覆盖。

### 改动
**文件**：`quarkdrive-webdav/src/vfs.rs` 方法 `fn rename()` 的 move 分支（约 line 811）。

**Before（无守卫）**：
```rust
let mut file = self
    .get_file(from.clone())
    .await?
    .ok_or(FsError::NotFound)?;
is_dir = file.dir;
// ...
self.drive
    .move_file(&file.fid, &to_parent_file.fid)
    .await
```

**After（空 fid 重试解析真实 fid）**：
```rust
let mut file = self
    .get_file(from.clone())
    .await?
    .ok_or(FsError::NotFound)?;
// 空 fid 守卫：上传提交后窄窗口内云端可能短暂返回空 fid，
// 直接 move_file("") 会 400 / 静默 no-op。重试解析父目录真实列表。
if file.fid.is_empty() {
    for _attempt in 0..3 {
        self.dir_cache.invalidate_parent(&from).await;
        if let Some(f) = self.get_file(from.clone()).await? {
            if !f.fid.is_empty() {
                file = f;
                break;
            }
        }
        tokio::time::sleep(std::time::Duration::from_millis(800)).await;
    }
    if file.fid.is_empty() {
        return Err(FsError::NotFound);
    }
}
is_dir = file.dir;
// ...
self.drive
    .move_file(&file.fid, &to_parent_file.fid)
    .await
```

## 10. MOVE 跨目录 400 — MoveFileRequest 缺必填字段（2026-07-08）

### 现象
真实 Finder 拖放"在挂载盘内拖动文件到子目录失败"；日志：
```
ERROR ... move file failed ... error=HTTP status client error (400 ) for url (.../file/move...)
```
此前 §9 仅加了空 fid 守卫，但实测跨目录 MOVE 仍 400。

### 根因
`MoveFileRequest` 只有 `filelist` + `to_pdir_fid` 两个字段，缺 Quark `file/move` 必填字段。
带有效 fid 调用仍 400；把 `post_request` 的错误响应体打到 error 级后，Quark 返回：
```json
{"status":400,"code":23027,"message":"不支持移动至系统创建文件夹"}
```
- 首要根因：请求体缺 `use_job_scheduler`（及 `exclude_fids`），Quark 直接 400 Bad Request。
- 次要：Quark 平台规则——**禁止 MOVE 进系统创建文件夹**（我的备份 / 来自：分享 / 快传 / 云解压 等）；
  `code=23027` 即此限制。上传/新建进系统文件夹是允许的（实测 PUT 进 /我的备份/ → 201），
  仅 MOVE 进系统文件夹被拒。

### 改动
- `drive/model.rs` `MoveFileRequest` 增加 `use_job_scheduler: bool`、`exclude_fids: Vec<String>`。
- `drive/mod.rs` `move_file` 填充 `use_job_scheduler: false, exclude_fids: vec![]`。
- `vfs.rs` `rename` move 分支补回 §9 的空 fid 重试守卫（当前源码缺失，跨目录移动在窄窗口内仍可能 400）。
- `drive/mod.rs` `post_request` 错误分支把响应体由 `debug!` 提升为 `error!`，便于以后直接看到 Quark 业务错误码。

### 验证（build_deploy_test.sh 部署后实测，HTTP 直连 8443）
1. MKCOL 用户子目录 → 201；上传文件 → 201；MOVE 进该用户子目录 → **201**；子目录 PROPFIND 200 含该文件，根目录不再含 → DELETE 文件/目录均 204。✓
2. 并发 2 文件上传根目录 → 均 201，均落盘（DELETE 204）。✓
3. MOVE 进系统文件夹 /我的备份/ → Quark 23027（平台限制，非本程序 bug）；上传进 /我的备份/ → 201 落盘（符合预期）。

### 经验
- Quark `file/move` 与 `file/copy` 必带 `use_job_scheduler`，否则 400；不要只靠空 fid 守卫。
- 系统文件夹的 MOVE 限制是平台行为，应在客户端对"移动目标为系统文件夹"给出明确提示，而非 500。
- `post_request` 失败时必须打印响应体（业务错误码在 body 里，不在 HTTP status text），否则只能看到裸 400。

## 11. Error -36（Finder 拖放上传"can't be read or written"）— 0字节 probe 跳过路径丢失 active_writes 注册（2026-07-08）

### 现象
用户拖放 PDF 文件（"领先的密码_BLM方法论.pdf"，约 4MB）到 `/Volumes/LocalQuark`，Finder 复制进度窗口显示"上传了一部分"，最终报 **Error -36**（"can't be read or written"）。日志完整时序：

```
04:46:45.095  Probe PUT (CL:0)     → 201  (0字节跳过)
04:46:45.855  Probe PUT (CL:0)     → 201  (又跳过)
04:46:45.857  LOCK                → 200  (proxy mocked)
04:46:46.385  PROPFIND verify      → 🔴404  ← Error -36 触发点
04:46:47.923  PROPFIND retry       → 🔴404
04:46:48.976  Content PUT (4MB)    → 201  (太晚了，Finder 已放弃)
```

### 根因
macOS `webdavfs_agent` 的 Finder 拖放序列是：
1. 空探针 PUT（CL:0，测试目标是否可写）
2. LOCK / UNLOCK
3. **PROPFIND 验证文件是否存在**
4. 真实内容 PUT

`do_flush()` 的 0 字节跳过守卫（§6 引入，防止创建 0 字节幽灵文件）在 `size==0 && old_fid.is_none()` 时：
- ✅ 正确跳过了 `up_pre`（不上传空文件到云）
- ❌ 但调用了 `after_flush()` → 从 `uploading` 列表移除条目 + 重置状态
- **❌ 从未调用 `register_active_write()`**

结果：文件不在云上、不在 active_writes overlay、不在 uploading 列表 → PROPFIND 三路查找全部落空 → 返回 404 → Finder 判定操作失败 → Error -36。

### 改动
`vfs.rs` `do_flush()` 第 1051 行：将 `self.after_flush().await?` 替换为 `self.fs.register_active_write(&pp, &self.file.file_name, 0u64, "").await`——注册一个 size=0 的占位 active_writes 条目。

效果：
- PROPFIND 验证步骤通过 active_writes 找到文件 → 返回 207 Multi-Status（含 size=0 + 有效 etag）
- Finder 继续发送真实内容 PUT → 覆盖占位条目为正确大小
- 45s 后占位条目自动过期清理

### 验证
Python 脚本精确模拟 Finder 序列（PROPFIND→probe PUT×2→LOCK/UNLOCK×2→验证 PROPFIND→content PUT→final PROPFIND→DELETE）：

| 步骤 | 修复前 | 修复后 |
|------|--------|--------|
| Probe PUT (CL:0) | 201 | 201 |
| LOCK / UNLOCK | 200 / 204 | 200 / 204 |
| **验证 PROPFIND** | **🔴 404** | **✅ 207** (含 getcontentlength=0 + etag) |
| Content PUT (1MB) | — (Finder 已放弃) | 201 |
| Final PROPFIND | — | 207 |
| DELETE | — | 204 |

部署版本：二进制 mtime Jul 8 12:53:27。

### 经验
- **active_writes overlay 是 Finder 可见性的生命线。** 任何让文件"暂时存在但不在云上"的代码路径都必须注册 active_writes 条目，否则 PROPFIND 立刻返回 404。
- `after_flush()` 是清理函数（删临时文件、移除 uploading、重置状态），**不是注册函数**。在跳过上传的 early-return 路径中调用它会导致所有追踪状态丢失。
- macOS webdavfs_agent 的拖放序列有严格的**验证步骤**（LOCK 后立刻 PROPFIND），容忍窗口 <1ms。任何延迟注册都会触发 Error -36。
- 诊断此类问题的方法：**完整 HTTP 时序日志 + 精确 Python repro**。GUI 不可复现时必须用脚本模拟每个请求头和时间间隔。

## 12. 替换（覆盖同名文件）第一次失败、之后成功 — 一致性窗口 + size-skip 误判（2026-07-08）

### 现象
Error -36 修复后，用户拖放上传成功，但**替换（覆盖同名文件）第一次失败、之后都成功**。
实测复现（Python 模拟 Finder 序列，8MB 文件，刚上传后 0.5s 内立刻覆盖）：

```
FRESH UPLOAD ow2_xxx.bin (8MB)      -> drag 204, GET match=True
OVERWRITE #1 (不同内容 8MB)         -> drag 204, GET match=False  ← 内容错误！
OVERWRITE #2                        -> drag 204, GET match=True
OVERWRITE #3                        -> drag 204, GET match=True
```
第一次覆盖"成功"(204) 但**下载内容是旧内容**（大小正确 8MB，字节不匹配）。无重名文件（name_hits=1，无 `(1)` 重复），所以是**内容被静默跳过上传**，不是重名冲突。

### 根因（三层叠加）
1. **`skip_upload_same_size` 误判（主因）**：`do_flush()` 中 `if self.fs.skip_upload_same_size && self.file.size == size` 在**覆盖**时也生效。第一次覆盖发生在刚上传后的云列表一致性窗口内，`open()` 通过 `active_writes` 自愈拿到的是**占位大小（旧文件 8MB）**，而新文件恰好也是 8MB → 误判"大小相同无需上传" → 跳过上传 → 旧内容残留。#2/#3 时 active_writes 已过期/更新，拿到真实云文件大小，故正常。
2. **覆盖探针破坏性（次因）**：原 0 字节跳过守卫是 `size==0 && old_fid.is_none()`。覆盖时 `old_fid` 存在，探针走 `up_pre` 的 0 字节「秒传」→ **删除真实文件并建 0 字节占位**（破坏性）。
3. **一致性窗口遮蔽（防卫性）**：`get_file` 只查缓存，刚上传后云列表未收敛时返回 None，`open()` 的 `active_writes` 自愈把已存在文件当成新建（fid=""），若此时上传会产生重名冲突。

### 改动
`vfs.rs` `do_flush`：
- **P2-3 解析块**（在捕获 old_fid 前）：当 `size>0 && self.file.fid.is_empty()` 且该路径近期有 `active_writes` 记录时，强制 `dir_cache.invalidate` + `get_or_insert` 重新列举父目录，找回真实同名旧文件的 fid 并作为 `old_fid`（覆盖语义）。消除一致性窗口内的"新建"误判。
- **0 字节跳过守卫** 由 `size==0 && old_fid.is_none()` 改为 `size==0`：覆盖探针也跳过 `up_pre`，不再删除真实文件（非破坏性）。
- **`skip_upload_same_size` 加 `old_fid.is_none()` 守卫**（P2-4）：仅对真正新建文件生效；覆盖（old_fid 存在）一律重新上传。内容完全相同的重拖仍由上方 `get_file_md5` 秒传/跳过正确短路。

### 验证（build_deploy_test.sh 部署后实测，HTTP 直连 8443）
`/tmp/repro_overwrite2.py`（8MB，含三种替换形态 + GET 内容校验 + Depth-1 重名检测）：

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 刚上传后立刻覆盖 x3（不同内容） | #1 GET match=False | 全部 match=True，name_hits=1，无重复 |
| 相同内容覆盖（md5-skip） | — | match=True ✅ |
| DELETE 后重传（Finder Replace 模式） | — | match=True ✅ |

部署版本：二进制 mtime Jul 8 13:07:32。

### 经验
- **覆盖必须用 `old_fid` 区分**：`skip_upload_same_size` / 任何"相同即跳过"优化都只能在 `old_fid.is_none()`（真正新建）时启用。覆盖时用户就是要替换（即使大小相同），必须重新上传。
- **`self.file.size` 在覆盖首窗口不可信**：它来自云端/active_writes 占位，可能滞后。判断"是否跳过上传"不能依赖它，必须依赖内容 md5（已存在文件的真实 md5）。
- **active_writes 自愈是把双刃剑**：它让上传中/刚上传的文件对 Finder 可见，但也会在一致性窗口内把"已存在文件"误判为"新建（fid=""）"。凡 `fid` 为空且近期有 active_writes 的路径，上传前必须重新向云确认真实旧文件。
- 0 字节探针对覆盖同样是非破坏性跳过，不要因为 `old_fid` 存在就走上传路径。
